# Agent notes

This repo hosts `app/`, a Bun/Electrobun/React desktop app, and `acp-mock-server/`, active shared test infrastructure for it. There is no native macOS client — see [ADR 0002](docs/adr/0002-revert-to-electrobun.md) (superseding ADR 0001) for the history of a native Swift/SwiftUI detour that was fully reverted. See `docs/ARCHITECTURE.md` and `docs/DESIGN.md`.

## Setup & running

Bun is used uniformly for dependency installation and running (`bun install` populates `node_modules`; `bun run dev`, `bun run dev:mock`, and `bun test` all use Bun as the runtime). pnpm is not used anywhere in this repo.

```bash
cd app
bun install
bun run dev:mock   # runs against acp-mock-server instead of the real Devin CLI
```

## Verification

From `app/`, these should succeed after changes there:

```bash
bun run typecheck
bun test
bun run build:macos-effects
bun run build:web
```

Avoid actually launching the GUI (`bun run dev`/`bun run start`) unless asked — it opens a real window.

From `acp-mock-server/`, these should succeed after mock server changes:

```bash
bun install
bun run typecheck
bun run build && node --test dist/tests/*.test.js
```

## Gotchas

- `acp-mock-server/tests/*.test.ts` are written against Node's built-in `node:test` APIs (`describe`/`test`/`beforeEach`/`afterEach` imported from `"node:test"`), not `bun:test`. CI's `acp-mock-server` job and `docs/RELEASE.md` both run `bun test` directly, which relies on Bun's `node:test` compatibility shim; that shim is version-sensitive — locally observed failing on Bun 1.3.14 (`NotImplementedError: describe() inside another test()` plus three 5s timeouts in `tests/subprocess.test.ts`) even though CI pins Bun 1.3.11. If `bun test` fails locally, don't assume a real regression before falling back to the suite's actual `package.json` `test` script — `bun run build && node --test dist/tests/*.test.js` — which exercises the same 15 tests directly against compiled output and isn't subject to Bun's shim.
- Keep `app/src/bun/acp/` provider-agnostic. Backend-specific (Devin, omp) process/permission-mode logic belongs in `app/src/bun/agent/runtime.ts` and the session adapter in `app/src/bun/index.ts`; the selected `AcpProviderId` is read from `AgentRuntimeContext.acpProvider`, not re-derived per call.
- `app/src/mainview/lib/electrobun.ts` constructs `new Electroview(...)` at module scope, which requires the native Electrobun webview bridge (`window.__electrobun`, injected by the native shell) to exist. Loading the built webview in a plain browser tab (e.g. via `bun run dev:hmr`'s standalone Vite dev server at `localhost:5173`, opened directly rather than through the Electrobun shell) throws `Cannot set properties of undefined (setting 'receiveMessageFromBun')` before React ever mounts. Automated visual verification (e.g. Playwright) against that plain dev server will only ever see a blank page — it can confirm the CSS/token pipeline loads (background color, etc.) but not render the app. Real visual verification needs the actual Electrobun shell (`bun run dev` / `bun run dev:mock`, `LEVEL5_USE_ACP_MOCK=1`), which isn't screenshot-able via a normal browser automation tool since it's a native WKWebView-hosted window, not a page in an ordinary browser context. Pure logic extracted out of files that import `electrobun.ts` (transitively or directly) — e.g. `app/src/mainview/diffFormat.ts` — is testable under plain `bun test`; anything that stays in a file importing that bridge is not.
- `app/src/mainview/index.css` and `docs/DESIGN.md` are the canonical, only source of truth for the design system (colors, type scale, spacing/radius, elevation, icon map in `lib/icon-map.ts`) — there is no separate native implementation to keep in sync with anymore.
- `app/assets/icon.png`, `assets/App.icns`, and `src/mainview/assets/app-icon.png` are regenerated from `app/assets/AppIconSource.png` (not designed independently). `scripts/apply-macos-icon.ts` only copies a pre-built `App.icns` into the built app bundle post-build, it does not generate one — regenerate `.icns` yourself via `iconutil` on macOS when the source art changes (see `docs/ARCHITECTURE.md`'s "App icon packaging" for the exact commands).
- New `app/src/bun/persistence/` stores that need durable storage should add their own migrations to `database.ts`'s ordered migration list and share its single `bun:sqlite` connection at `~/.level5build/level5.sqlite`, rather than opening a separate connection; see `RecentProjectStore` for the pattern.
- Switching the ACP provider setting (`setAcpProvider` RPC) does not restart or otherwise touch any already-running `ProjectAgentConnection`. There's no explicit "restart all connections" call. `ensureProcess` picks up the change on its own the next time it runs for that project: the freshly-computed backend id no longer matches the connection's cached `processPermissionMode` (the same string used for cwd/approval-mode-change detection), so the existing mismatch-triggers-respawn path fires and the new provider takes over transparently. Don't add a special-case provider-change teardown path — it would duplicate logic that already exists for this exact purpose.
- There is no ACP `session/list` call anywhere in `app/src/bun/index.ts`; the sidebar is sourced entirely from the SQLite-backed session cache (`listAllSessionRows`), and `session/load` only ever runs as a silent send-time "prime" (never on selection). Don't reintroduce a discovery RPC call to "fix" a sidebar gap — a backend's own (potentially much larger, cross-app) session history must never bleed into the sidebar.
- A "New chat" is not persisted to SQLite (and does not appear in the sidebar) until the first message is actually sent. `ensureSession`'s warm-up path (project selection, composer priming) passes `persist: false`; only `startPrompt`'s actual send persists, even when reusing an already-warmed session. Don't make warm-up eagerly persist "for consistency" — it would make every project selection create a phantom sidebar row.
- Real Devin's `session/load` requires both `mcpServers` and `cwd` in its params, or it's rejected outright ("missing field cwd"). Always pass the project's own key as `cwd` (a project's key *is* its normalized cwd) — don't pass `cwd: null` even though the mock server ignores it.
- Killing a project's `devin acp` process (approval-mode restart, or app quit) without first calling `session/close` on every session that process created/primed permanently orphans those sessions server-side ("already open in another process" on the next reconnect, even from that same project's own next relaunch). `app/src/bun/index.ts`'s quit teardown (`teardownAgentClientAndExitSafely`) exists specifically to run this close-before-kill cleanup before the process actually exits — don't remove it or bypass it with a raw `process.exit()`/`kill` elsewhere.
- `ProjectAgentConnection.withConnectionSetupLock` (`app/src/bun/index.ts`) serializes `ensureProcess`/`ensureInitialized`/session-creation across `startPrompt`, `prepareSession`, and `bestEffortDeleteSession` for the same project. Real Devin's ACP server does not handle an unrelated `session/new`/`session/load` racing a real prompt against the same underlying process safely — don't bypass the lock by calling `ensureProcess`/`ensureInitialized` directly.
- `ACP_REQUEST_TIMEOUTS_MS.prompt` (`app/src/bun/acp/client.ts`) must stay a generous explicit override (currently 6h), never the transport's short default meant for quick RPCs like `initialize`/`session/new`. Unlike those, `session/prompt` doesn't resolve until the *entire* turn completes, which routinely exceeds a short default for any real tool-using turn. Binding it to a short default previously caused a live, reproduced-in-production bug: the request times out and gets reported as "Prompt failed" while the agent keeps working; its `session/update` notifications (and even the eventual real `session/prompt` response) still arrive but the response is now for a request nobody's waiting on anymore, silently dropped. Detecting a truly stuck turn is the turn-idle watchdog's job (resets on every inbound event) — that's the real timeout mechanism; `prompt`'s own timeout is just a last-resort safety net against a leaked continuation.
- `app/src/bun/git/status.ts`'s `runGitCommand` has a short (3s) timeout on every spawned `git` command so a hung repository (e.g. an unresponsive credential helper) can't hang the dashboard indefinitely. Keep new git subprocess calls routed through it rather than spawning `git` directly.
- `app/src/bun/git/review.ts`'s `git diff --numstat` call must keep the `-z` flag, and `parseReviewNumstat` must keep parsing NUL-delimited fields rather than newline-delimited text. Without `-z`, a file that is both renamed and edited above git's similarity threshold prints as a single compact line (`6\t6\tsrc/{foo.txt => bar.txt}`) with no unambiguous path field, silently orphaning that file's stats from `enrichFile`'s lookup by the porcelain-derived `file.path` (additions/deletions render as 0 despite real changes). This isn't an edge case — it's git's default output for any rename+edit, same-directory or cross-directory. See `review.test.ts`'s "assigns numstat additions/deletions to a renamed-and-edited file's new path" test.
- A `session_info_update` ACP notification can arrive unprompted right after `session/new`, before any message was ever sent (the mock server's post-create timer does this; real Devin plausibly does too). `ProjectAgentConnection.handleNotification`'s `session_info_update` branch (`app/src/bun/index.ts`) must gate on `shouldPersistSessionInfoUpdate` (`app/src/bun/agent/runtime.ts`: notification's session id matches the connection's current session AND `sessionPersisted` is already true) before calling `rememberSession` — otherwise this push path bypasses the "no sidebar row until first send" invariant (see the gotcha above) through a side door `ensureSession`'s `persist` flag never sees.
- `AgentAcpClient.listConfigOptions()`/`listSlashCommands()` (`app/src/bun/index.ts`) must `await` the target connection's `awaitSetup()` (which resolves `connectionSetupChain`, the same promise `withConnectionSetupLock` serializes on) before reading `configOptions`/`slashCommands`, not sample them instantly. Composer config population is a race against the backend's own process spawn + `initialize` + `session/new` handshake, which takes real wall-clock time against a real backend (`devin`/`omp`; observed 0.6-3s+) but resolves near-instantly against the mock server. A synchronous read that "worked" under `dev:mock` reliably lost that race against a real backend and left the model selector permanently empty — this only reproduces against real Devin/omp, not the mock server, so don't treat a clean `dev:mock` run as proof the composer's model/slash-command population actually works.
- `startPrompt` (`app/src/bun/index.ts`) is the sole, authoritative writer of the user's own prompt into the transcript (`kind: "message", role: "user"`, persisted synchronously before `session/prompt` is dispatched) — it must never go back to relying on the ACP process echoing the prompt back as a `user_message_chunk` notification to trigger persistence. Real Devin/omp has been observed to never send that echo, which previously meant no user-authored message ever made it into `~/.level5build/level5.sqlite`: it was visible for the live session only via the renderer's local optimistic bubble (`dispatchPrompt`/`optimisticUserTextRef` in `App.tsx`), then silently gone on relaunch since nothing had ever persisted it. `handleNotification`'s `user_message_chunk` branch is now an intentional no-op for exactly this reason — don't reintroduce persistence there, it would race or duplicate the authoritative write.
- `upsertTranscriptUpdate`'s `message`/`thought` merge branches (`app/src/bun/index.ts`) must only fall back to "append onto whatever entry is last" when the incoming update carries no `messageId` at all (some backends omit it on streamed chunks, so adjacency is the only correlation signal available). Falling back whenever an *exact* `messageId` match merely fails — rather than checking it's absent — silently concatenates a genuinely new, distinct message (e.g. `startPrompt`'s freshly generated `crypto.randomUUID()` per send) onto an unrelated prior entry of the same role/kind. This is reachable in practice: a turn that errors before any agent reply streams reports its failure via `emit()` (never persisted), so the transcript's last entry stays the previous user message, and the next send's synthesized message would merge into it instead of appending as its own row.
